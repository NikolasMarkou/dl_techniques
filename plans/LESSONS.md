## Process

- **"Remove X" can mean two different things.** When a user asks to remove a multi-stage / multi-mode surface, distinguish between (a) the *capability* is unwanted and (b) the *naming or layering* is confusing. EXPLORE should classify each piece before PLAN drafts deletion steps.
- **Doc updates belong in the same plan as the code change they describe.** Treating README/CLAUDE.md updates as out-of-scope follow-ups means the plan is technically "complete" while the user-visible behavior (CLI flags, examples) is still wrong. If a doc is the only public-facing reference to flags or APIs being changed, include it as a plan step from the start.
- **Static grep gates can substitute for runtime probes — but only for purely mechanical surface changes.** Renames with no internal recursion and no out-of-file callers are safe to verify with `grep -n`. Anything that touches behavior (loss math, optimizer wiring, dataset paths) needs a runtime smoke test.
- **PLAN -> PLAN cycles are normal and not a failure.** Better to revise the written plan once after user feedback than to silently re-interpret the goal during EXECUTE.
- **Mechanical pattern-replication across N files is a valid one-iteration plan when each site has been individually characterised in EXPLORE.** Replicating a known-good fix across N sibling training scripts closes in a single iteration with zero fix attempts when EXPLORE catches the outliers BEFORE PLAN. Lesson: when copying a precedent across N sites, EXPLORE must per-file diff the precedent's structural assumptions (return type, surrounding control flow, presence of related setup) — outliers found at EXECUTE time turn one-iter plans into pivots. Resist extracting a shared helper for <=4 sites if extraction crosses package boundaries; per-file repetition keeps each callback grepable.
- **Pure additive layer/class work fits a single iteration of the protocol.** EXPLORE -> PLAN -> EXECUTE -> REFLECT closes in one session when the change is non-destructive (sibling class, no signature changes to existing public surface) and tests are scoped to one file.
- **Sibling-stack model additions: prefer parallel package over factory retrofit when the new component has a different mask shape or unique hyperparameters.** Build a self-contained decoder stack inside the new model package instead of modifying the shared factory + transformer block. Trade ~150 LOC duplication for zero blast radius on 30+ unrelated models.
- **Orchestrator-crash mid-EXECUTE recovery: trust git log + filesystem, not stale state.md.** When a session crashes mid-EXECUTE while updating state.md, the file may say "Step N" but git log shows steps N+1, N+2 already committed. Recovery: enumerate `git log` since baseline, diff vs plan.md "Files To Modify", re-run pytest to confirm green, then write a "Resume Notes" section in state.md and proceed to REFLECT. Do NOT re-execute completed steps.
- **Plan-time line-count predictions for sibling-class additions underestimate by ~2x.** "Mirror the parent class" tends to count algorithmic body only, missing validation, docstring, `get_config`, and `compute_output_shape`.
- **Plan-time line-count predictions underestimate by 5-10x for plans that touch dtype semantics under mixed precision OR span >3 consumer scripts.** Bug fixes in dtype semantics pull in symbolic-graph plumbing changes (e.g. `compute_output_spec` overrides) that aren't visible at EXPLORE time. Plans touching multiple consumer scripts undercount because each consumer absorbs ~50 lines of CLI plumbing + dataclass fields + helper wiring.
- **Pre-existing tests can encode bugs as contracts.** When a fix to a library bug causes adjacent tests to fail, read the failing assertions before assuming regression — they may be locking in the buggy behavior. Update asserts and anchor the new contract with a `# DECISION` comment so the next reader doesn't re-introduce the bug.
- **Empirical confirmation of a finding does not separate bug from contract.** A "deep review" agent can confirm a behavior occurs without checking whether that behavior is *enforced by an existing test as a deliberate contract*. Before planning fixes for review-flagged "silent" / "wrong" behavior, grep the test suite for tests that *lock in* the current behavior.
- **Run the existing test suite as the FIRST step of EXPLORE when the goal is "fix issues found in a review".** All-green tests are the cheapest evidence that some review claims are wrong.
- **Internal-consistency arguments are not falsification.** A claim of the form "X is causal because Y" / "X is correct because Z" must have a unit test on X *alone* before it gates an end-to-end design. Add a micro-test on the falsifiable building block as part of the plan, not as a follow-up.
- **Verify reviewer claims empirically before applying fixes from a "deep review".** Reviews contain false positives. For each "critical" claim about API behavior (e.g. "keras.ops.rfft returns complex"), run a 3-line empirical check first. Lock empirically-correct behavior in a test (e.g. `assert isinstance(kernel_fft, tuple)`) so future "fixes" surface as test failures.
- **A correctly-flagged bug can mask a worse adjacent bug.** When a reviewer flags one mixed-precision/dtype/serialization issue, sweep the whole category before merging the narrow fix — the broader issue often dominates. Add an end-to-end test for the broader category to lock it.
- **Verify "every site does X" with grep BEFORE PLAN, not during EXECUTE.** Textual claims in findings (e.g. "all 5 probe classes implement `_post_generate_hook`") that look like inventory facts but are unchecked have a way of being half-true. EXPLORE-time `grep -rn 'X' src/train/ | wc -l` is cheap insurance — costs one tool call, prevents a mid-EXECUTE finding correction. Plan plan_3f461682 paid for skipping this with one in-stream D-002 (3 cliffordnet probes lacked the hook; resolution was +9 LOC each, no harm but avoidable).
- **Per-trainer "1-line bind" of a callback hook is actually a 4-line delta.** Pattern `callbacks.append(GenerationProbeCallback(...))` must become `probe_cb = GenerationProbeCallback(...); probe_cb._hook = fn; callbacks.append(probe_cb)`. Predict 4 LOC per bind-site in plan budgets, not 1.

## Codebase-specific (dl_techniques)

- **Do not run `make test` as a regression check.** Full suite is ~1.5h and is the pre-push hook. Scope pytest to changed modules only.
- **Use `MPLBACKEND=Agg` for any training-script invocation.** Headless servers crash on the default matplotlib backend.
- **Use `dl_techniques.utils.logger` only — no `print` calls in library or training code.** Hard convention; grep for `print(` in modified files.
- **Single GPU jobs only.** Never spawn parallel training runs (memory + driver contention on the dev box).
- **Pin GPU via shell env, not `setup_gpu(args.gpu)` alone.** TensorFlow initialises GPU devices on `import tensorflow as tf`, which happens at module top — *before* `main()` runs `setup_gpu()` and sets `CUDA_VISIBLE_DEVICES`. Workaround: prefix the command, e.g. `CUDA_VISIBLE_DEVICES=1 MPLBACKEND=Agg python -m train.<...>`.
- **GPU-OOM during EXPLORE/REFLECT verification: fall back to CPU with `CUDA_VISIBLE_DEVICES=""`.** Metric ops built on `keras.ops.*` are backend-agnostic; unit-test verification doesn't need GPU. Used during plan_3f461682 REFLECT — 170/170 metric tests green on CPU in 8.84s.
- **CLI flag uniformity across sibling training scripts is an explicit design property.** Once one consumer in a script family adopts a flag (e.g. `--steps-per-epoch`, `--seed`, `--shuffle-shards`), all siblings should — otherwise users learn N slightly-different CLIs. Encode this expectation in the docstring of the centralised helper.
- **For packed CLM, `steps_per_epoch = num_articles // batch_size` is wrong.** Packed CLM emits chunks of `max_seq_length` tokens, not articles. Use `train.common.nlp.estimate_clm_steps_per_epoch(num_articles, max_seq_length, batch_size, override=None, avg_tokens_per_article=440)`.
- **For Wikipedia CLM, the default `min_article_length=0` is correct.** The packed pipeline concatenates every token regardless of source article length; filtering stubs only loses tokens.
- **Smoke recipe for `train_clip.py`-class scripts:** `--synthetic --max-train-samples 64 --max-val-samples 32 --epochs 1 --batch-size 8` verifies CLI + dataset wiring + one full epoch end-to-end without external downloads.
- **Keras 3.8 `model.load_weights(path.keras, by_name=True)` is broken.** Use `dl_techniques.utils.weight_transfer.load_weights_from_checkpoint` for name-based transfer from `.keras` checkpoints.
- **AdamW double weight decay is a footgun.** Never combine `AdamW(weight_decay=...)` with `kernel_regularizer=L2(weight_decay)` — pick one. Also: exclude `logit_scale` from AdamW weight decay in CLIP-style training.
- **`CliffordNetBlock` is dim-preserving.** Hierarchical CliffordNets must use explicit downsamplers. `CliffordNetBlockDS` (vision) and `CausalCliffordNetBlockDSv2` (language, causal) integrate downsampling INTO the block.
- **CliffordNet does not tolerate aggressive patchify stems.** patch=4 stems collapse spatial structure before any CliffordNetBlock runs, costing 6+ pp val_acc on CIFAR-100. Keep stem stride <= 2 before the first block stack.
- **On CIFAR-100 with ~3M params at 100 epochs, downsampler choice is ~ noise.** Strided-conv / avgpool+1x1 / Swin patch-merging converge within <0.7 pp.
- **GPU 1 (RTX 4070 12GB) caps CliffordNet training at batch 64 for hierarchical variants.** Halve batch when switching from GPU 0.
- **For shape-preserving Clifford block variants: pool BEFORE the stream split.** The geometric product is element-wise on (z_det, z_ctx); their spatial+channel dims must match.
- **`DepthwiseConv2D` on TF GPU rejects asymmetric strides.** `DepthwiseConv2D(strides=(1, s))` with `s != 1` fails. Substitute `Conv2D(filters=channels, kernel_size=(1, k), strides=(1, s), groups=channels)`.
- **For causal `(H=1, W=seq_len)` Clifford blocks, pool surface MUST be restricted to `avg`/`max` only.** Other variants leak future info. `padding="same"` on `AveragePooling2D` / `MaxPooling2D` IS causal.
- **Pool -> nearest-upsample round trip is NOT causal at fine resolution — only at the pool grid.** Mitigation: causal right-shift by `s-1` along W after each `UpSampling2D`. Implemented as 4-line `_causal_upsample(x, s)` in `cliffordnet/lmunet.py`.
- **`.keras` save/load round-trip on GPU under fp32 has reduction-order noise ~5e-5 for U-Net-shaped models** (decoder concat + 1x1 projection compounds reduction noise across levels). Default unit-test tolerance for U-Net-shaped models should be 1e-4, not 1e-5.

## CLM training metrics & probes

- **Metric *math* belongs in `dl_techniques.metrics/`; the per-trainer-shape *metric list* belongs in `train.common.nlp`.** New plan plan_3f461682 split `BitsPerToken` / `BitsPerCharacter` (math) into `dl_techniques/metrics/llm_metrics.py` and `build_clm_metrics(encoding_name, ignore_index, chars_per_token=None)` (list, with per-encoding `_CHARS_PER_TOKEN_DEFAULTS`) into `train/common/nlp.py`. The split keeps the library reusable for non-training callers and lets the training-side helper centralize ignore-index propagation across `MaskedCausalLMLoss` (-1) vs HF-style (-100).
- **`build_clm_metrics()` MUST return fresh metric instances on every call.** Keras `model.compile()` requires unique state per metric per compile. Test it: `a = build(); b = build(); assert a[i] is not b[i]`.
- **For new CLM metrics, set `dtype="float32"` explicitly on `add_weight` accumulators.** AMP isn't currently enabled in CLM trainers, but the metric should remain valid if a future plan flips `mixed_float16` on. The existing `Perplexity` already does this implicitly via Keras default; new metrics should be explicit.
- **Probe-time aggregate metrics (Self-BLEU, distinct-N, mean tok/s) ride on `_post_generate_hook`, not a new probe subclass.** The probe class is duplicated 5x; subclassing per trainer would compound the duplication. A free function bound by `probe_cb._post_generate_hook = fn` is cleaner. Make the function schema-tolerant (`.get(...)` reads, log-and-skip on shape error) so it survives the routing-variant probe's `Tuple[str, int]` `_generate` return without modification.

## Keras 3 layer / model composition gotchas

- **Frozen state inside a layer composed into a `keras.Model` MUST live in `add_weight(trainable=False, ...)` OR be stored as numpy on `self` and converted inside `call()` — never as a plain tensor created in `build()`.** Keras runs `compute_output_spec` in a transient scratch `FuncGraph`; `build()` runs there too, and any `ops.stack(...)` / `ops.convert_to_tensor(...)` result gets captured by that graph. Subsequent eager calls dereference a dead tensor. Two safe options: (a) `add_weight(name=..., shape=..., initializer=keras.initializers.Constant(numpy_value), trainable=False)`; (b) store numpy on `self` and call `ops.convert_to_tensor(self._x_np, dtype=...)` inside `call()`.
- **Symptom-only manifestation:** invisible when the layer is exercised standalone; tests that wrap it in a `keras.Model` fail. Always include a "wrap in `keras.Model` + `model.save()` round-trip" test for any new layer with frozen state.
- **`compute_output_spec` declares symbolic output dtype independently from `call()`'s runtime return.** Under `mixed_float16`, the Functional graph coerces returns back to fp16 to match the declared spec. If you need a layer to return a different dtype than `compute_dtype` from `call()`, override `compute_output_spec`.
- **Probability-distribution layers under fp16 mixed precision:** clip+sigmoid in fp16 underflows for vocabularies with leaves around or below the fp16 normal range. Cast logits to fp32 BEFORE sigmoid+clip, run tree accumulation in fp32. Scope override to fp16 (bf16 unaffected).
- **`add_weight` variables behave differently under AMP than activations.** Variables stay fp32 but get autocast to compute_dtype when read inside `call()`. Mixing this with explicit `ops.cast(..., "float32")` (e.g. of `arange` / index tensors) produces silent crashes at downstream `Mul`/`Einsum`. Resolution patterns: (a) fence sub-paths that need fp32 with explicit `ops.cast(weight, "float32")` reads, (b) cast index tensors to `self.compute_dtype` at the einsum boundary, (c) cast variable-derived tensors to the activation dtype just before mixing. Add a `@pytest.mark.parametrize("policy", ["mixed_float16","mixed_bfloat16"])` end-to-end test to guard.
- **Custom Keras `Initializer` subclass beats split-weight refactors when test count is locked.** Replacing `np.random` inside `build()` is tempting via two `add_weight` calls (Identity + RandomNormal), but tests that assert `len(layer.trainable_variables) == N` lock the weight topology. A `class _MyInit(keras.initializers.Initializer)` with `__call__` + `get_config` keeps topology unchanged AND is fully serializable.

## Layer / model testing patterns (dl_techniques)

- **Test class structure for new layers: mirror existing sibling classes in the same file.** `tests/test_layers/test_geometric/test_clifford_block.py` is a strong reference — pytest fixtures for params, classes per layer, tests cover init/build/shape/cli_modes/numerical_stability/serialization/save_load/gradient_flow/stacking.
- **`test_residual_identity_at_init`** (gamma init ~ 0 -> output ~ input) is the cheapest sanity check that residual wiring is correct. For downsampling variants, the reference is `pool(input)` instead of `input`.
- **For pool-choice variants (avg vs max), verify they produce different outputs** with `layer_scale_init` near zero so the residual dominates.
- **Always include a `test_save_load_roundtrip` that wraps the layer inside a `keras.Model` and round-trips through `.keras` save/load.**
- **For causal models with downsample/upsample paths, write a unit micro-test on each spatial-mixing helper in isolation, in addition to the end-to-end causality test.** End-to-end perturbation tests can pass on small configs while masking a bug at deeper variants.
- **Causality test recipe for `(H=1, W=seq_len)` models:** (1) perturb-last-position, (2) perturb-middle-position with depth>=2, (3) non-multiple `seq_len % total_stride` (exercises pad+crop), (4) with `use_global_context=True` if applicable. Tolerance on earlier-position diff: `< 1e-5`.
- **For new metric classes (BPT/BPC/etc.), test against a reference implementation by identity** — e.g. `BitsPerToken == log2(Perplexity)` to within 1e-5 on the same synthetic logits/labels. This catches reduction-order bugs that perfect-prediction smoke tests miss.
