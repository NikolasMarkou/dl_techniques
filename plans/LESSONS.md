## Process

- **"Remove X" can mean two different things.** When a user asks to remove a multi-stage / multi-mode surface, distinguish between (a) the *capability* is unwanted and (b) the *naming or layering* is confusing. EXPLORE should classify each piece before PLAN drafts deletion steps.
- **Doc updates belong in the same plan as the code change they describe.** Treating README/CLAUDE.md updates as out-of-scope follow-ups means the plan is technically "complete" while the user-visible behavior (CLI flags, examples) is still wrong. If a doc is the only public-facing reference to flags or APIs being changed, include it as a plan step from the start.
- **Static grep gates can substitute for runtime probes — but only for purely mechanical surface changes.** Renames with no internal recursion and no out-of-file callers are safe to verify with `grep -n`. Anything that touches behavior (loss math, optimizer wiring, dataset paths) needs a runtime smoke test.
- **PLAN -> PLAN cycles are normal and not a failure.** Better to revise the written plan once after user feedback than to silently re-interpret the goal during EXECUTE.
- **Pure additive layer/class work fits a single iteration of the protocol.** EXPLORE -> PLAN -> EXECUTE -> REFLECT closes in one session when the change is non-destructive (sibling class, no signature changes to existing public surface) and tests are scoped to one file. Don't over-engineer; still write findings, plan, decisions, verification.
- **Plan-time line-count predictions for sibling-class additions underestimate by ~2x.** "Mirror the parent class" tends to count algorithmic body only, missing validation, docstring, `get_config`, and `compute_output_shape`. Pad the budget — bound is "matches sibling size" not "matches algorithmic delta".
- **Pre-existing tests can encode bugs as contracts.** When a fix to a library bug causes adjacent tests to fail, read the failing assertions before assuming regression — they may be locking in the buggy behavior. Update asserts and anchor the new contract with a `# DECISION` comment so the next reader doesn't re-introduce the bug.
- **Plan-time line-count predictions underestimate by 5-10x for plans that touch dtype semantics under mixed precision.** Bug fixes in this area pull in symbolic-graph plumbing changes (e.g. `compute_output_spec` overrides) that aren't visible at EXPLORE time because the symptom only manifests under Functional-graph composition.
- **Empirical confirmation of a finding does not separate bug from contract.** A "deep review" agent can confirm a behavior occurs without checking whether that behavior is *enforced by an existing test as a deliberate contract*. Before planning fixes for review-flagged "silent" / "wrong" behavior, grep the test suite for tests that *lock in* the current behavior.
- **Run the existing test suite as the FIRST step of EXPLORE when the goal is "fix issues found in a review".** All-green tests are the cheapest evidence that some review claims are wrong.
- **Internal-consistency arguments are not falsification.** A claim of the form "X is causal because Y" / "X is correct because Z" must have a unit test on X *alone* before it gates an end-to-end design. The cliffordnet U-Net plan accepted "nearest-upsample is causal" on the strength of an internal derivation; the derivation was correct *only at the pool grid* and the end-to-end causality test failed at Step 3 smoke check (Pre-Mortem Scenario A fired). Add a micro-test on the falsifiable building block as part of the plan, not as a follow-up.

## Codebase-specific (dl_techniques)

- **Do not run `make test` as a regression check.** Full suite is ~1.5h and is the pre-push hook. Scope pytest to changed modules only.
- **Use `MPLBACKEND=Agg` for any training-script invocation.** Headless servers crash on the default matplotlib backend.
- **Use `dl_techniques.utils.logger` only — no `print` calls in library or training code.** Hard convention; grep for `print(` in modified files.
- **Single GPU jobs only.** Never spawn parallel training runs (memory + driver contention on the dev box).
- **Smoke recipe for `train_clip.py`-class scripts:** `--synthetic --max-train-samples 64 --max-val-samples 32 --epochs 1 --batch-size 8` verifies CLI + dataset wiring + one full epoch end-to-end without external downloads. Use `--skip-pretrain` (or equivalent) to keep the loop fast.
- **Keras 3.8 `model.load_weights(path.keras, by_name=True)` is broken.** Use `dl_techniques.utils.weight_transfer.load_weights_from_checkpoint` for name-based transfer from `.keras` checkpoints.
- **AdamW double weight decay is a footgun.** Never combine `AdamW(weight_decay=...)` with `kernel_regularizer=L2(weight_decay)` — pick one. Also: exclude `logit_scale` from AdamW weight decay in CLIP-style training, otherwise the temperature decays to zero and the contrastive signal dies silently.
- **`CliffordNetBlock` is dim-preserving.** Hierarchical CliffordNets must use explicit downsamplers. `CliffordNetBlockDS` (vision) and `CausalCliffordNetBlockDSv2` (language, causal) integrate downsampling INTO the block — prefer them at stage boundaries over external downsamplers.
- **CliffordNet does not tolerate aggressive patchify stems.** patch=4 stems collapse spatial structure before any CliffordNetBlock runs, costing 6+ pp val_acc on CIFAR-100. Keep stem stride <= 2 before the first block stack.
- **On CIFAR-100 with ~3M params at 100 epochs, downsampler choice is ~ noise.** Within a 3-stage 64->128->256 hierarchy, strided-conv / avgpool+1x1 / Swin patch-merging converge within <0.7 pp. Pick the simplest (avgpool+1x1) unless you have a reason.
- **GPU 1 (RTX 4070 12GB) caps CliffordNet training at batch 64 for hierarchical variants.** Batch 128 fits V0-isotropic but V4 (4-stage 512-ch) OOMs. Halve batch when switching from GPU 0.
- **For shape-preserving Clifford block variants: pool BEFORE the stream split.** The geometric product is element-wise on (z_det, z_ctx); their spatial+channel dims must match. Downsample `x_norm` once and feed both streams from the pooled tensor; use a separate pool instance for the residual skip on `x_prev` (cleaner serialization than reusing one layer).
- **`DepthwiseConv2D` on TF GPU rejects asymmetric strides.** `DepthwiseConv2D(strides=(1, s))` with `s != 1` fails. For stride-asymmetric depthwise patterns (e.g. `(H=1, W=seq_len)` causal layouts that downsample only along W), substitute `Conv2D(filters=channels, kernel_size=(1, k), strides=(1, s), groups=channels)` — algebraically identical, no GPU restriction.
- **For causal `(H=1, W=seq_len)` Clifford blocks, pool surface MUST be restricted to `avg`/`max` only.** `pyramid_diff` / `blur` / `gaussian_dw` / `pixel_unshuffle` / `resnetd` either leak future info under `padding="same"` or change channel semantics in ways the geometric product can't absorb. `padding="same"` on `AveragePooling2D` / `MaxPooling2D` IS causal (zero / -inf padding does not leak future). Test: perturb the last input position and assert earlier outputs are byte-identical at strides=2.
- **Pool -> nearest-upsample round trip is NOT causal at fine resolution — only at the pool grid.** DSv2 with `padding="same"` and stride `s` produces pooled cell `j` covering input positions `[j*s, j*s+s-1]`. Nearest-upsample maps output position `k` to pooled cell `k // s`, which has seen up to input `(k//s)*s + s - 1`. For `k % s != s-1` that maximum exceeds k — future leak. Mitigation: causal right-shift by `s-1` along W after each `UpSampling2D` (left zero-pad, drop right). Then output position k reads pooled cell `(k-(s-1))//s`, max input seen <= k. Implemented as 4-line stateless helper `_causal_upsample(x, s)` in `cliffordnet/lmunet.py` (D-007). Anchor with `# DECISION D-007` at definition and call site.
- **`.keras` save/load round-trip on GPU under fp32 has reduction-order noise ~5e-5 for U-Net-shaped models** (decoder concat + 1x1 projection compounds reduction noise across levels). Default unit-test tolerance for U-Net-shaped models should be 1e-4, not 1e-5. `lm.py`-style isotropic models can hold 1e-5; U-Net cannot. Cause: fp32 XLA reduction-order non-determinism, not a `get_config` bug.

## Keras 3 layer / model composition gotchas

- **Frozen state inside a layer composed into a `keras.Model` MUST live in `add_weight(trainable=False, ...)` OR be stored as numpy on `self` and converted inside `call()` — never as a plain tensor created in `build()`.** Keras runs `compute_output_spec` in a transient scratch `FuncGraph`; `build()` runs there too, and any `ops.stack(...)` / `ops.convert_to_tensor(...)` result gets captured by that graph. Subsequent eager calls dereference a dead tensor: `tensor cannot be accessed from here, it was defined in FuncGraph(...) which is out of scope`. Two safe options: (a) `add_weight(name=..., shape=..., initializer=keras.initializers.Constant(numpy_value), trainable=False)`; (b) store numpy on `self` (e.g. `self._mask_np = mask_np`) and call `ops.convert_to_tensor(self._mask_np, dtype=...)` inside `call()`. Option (b) keeps recomputable state out of checkpoints.
- **Symptom-only manifestation:** invisible when the layer is exercised standalone (no surrounding `compute_output_spec` graph). Tests that instantiate the layer directly pass; tests that wrap it in a `keras.Model` fail. Always include a "wrap in `keras.Model` + `model.save()` round-trip" test for any new layer with frozen state.
- **`compute_output_spec` declares symbolic output dtype independently from `call()`'s runtime return.** Keras' default `compute_output_spec` returns `KerasTensor(dtype=self.compute_dtype)` when a custom `compute_output_shape` is implemented. Under `mixed_float16`, even if `call()` casts the return to fp32, the Functional graph coerces it back to fp16 to match the declared spec. If you need a layer to return a different dtype than `compute_dtype` from `call()`, override `compute_output_spec` to declare the runtime dtype.
- **Probability-distribution layers under fp16 mixed precision:** clip+sigmoid in fp16 underflows for vocabularies with leaves around or below the fp16 normal range (~6.1e-5; 50K-vocab leaves ~1.5e-5). Cast logits to fp32 BEFORE sigmoid+clip, run tree accumulation in fp32, keep final output in fp32 under fp16 only (bf16 is unaffected). Scope the override to fp16.

## Layer / model testing patterns (dl_techniques)

- **Test class structure for new layers: mirror existing sibling classes in the same file.** `tests/test_layers/test_geometric/test_clifford_block.py` is a strong reference — pytest fixtures for params, classes per layer, tests cover init/build/shape/cli_modes/numerical_stability/serialization/save_load/gradient_flow/stacking. Copy that structure when adding a sibling class.
- **`test_residual_identity_at_init`** (gamma init ~ 0 -> output ~ input) is the cheapest sanity check that residual wiring is correct. For downsampling variants, the reference is `pool(input)` instead of `input`.
- **For pool-choice variants (avg vs max), verify they produce different outputs** with `layer_scale_init` near zero so the residual dominates. Otherwise random init noise drowns out the pool difference.
- **Always include a `test_save_load_roundtrip` that wraps the layer inside a `keras.Model` and round-trips through `.keras` save/load.** Layers that pass standalone tests but break under model composition are a recurring class of bugs in Keras 3 (see FuncGraph gotcha above).
- **For causal models with downsample/upsample paths, write a unit micro-test on each spatial-mixing helper in isolation, in addition to the end-to-end causality test.** End-to-end perturbation tests can pass on small configs while masking a bug at deeper variants; isolating each helper (e.g. `_causal_upsample`) makes regressions surface as a single failing micro-test rather than a confusing end-to-end failure. Pattern: perturb input at position k, assert output positions < expected-causal-limit are byte-identical.
- **Causality test recipe for `(H=1, W=seq_len)` models:** (1) perturb-last-position, (2) perturb-middle-position with depth>=2, (3) non-multiple `seq_len % total_stride` (exercises pad+crop), (4) with `use_global_context=True` if applicable. Tolerance on earlier-position diff: `< 1e-5`.
